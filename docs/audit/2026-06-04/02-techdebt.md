## [authz]

### [CRITICAL] Public /api/auth/register lets anyone self-provision a PARENT account into any family (cross-tenant privilege escalation)  — ✅REAL
- Domain: auth · Category: security · Effort: S
- Evidence: backend/app/api/routes/auth.py:43-53 `register()` has no auth dependency and is whitelisted public in frontend/src/middleware.ts:76. It calls AuthService.register_user (backend/app/services/auth_service.py:50-58) which sets `role=user_data.role` and `family_id=user_data.family_id` straight from the client body. Schema backend/app/schemas/user.py:24-29: `role: UserRole = UserRole.CHILD` and `family_id: UUID` are both client-controllable. The only guard is that the family exists (auth_service.py:42-47) — no check the caller belongs to it or is a parent. The backend is publicly reachable at api-gcp-family.agent-ia.mx, and the route is registered with prefix /api/auth (backend/app/main.py:150). The safe self-service path is /register-family; /register is the raw, dangerous one and is still live.
- Impact: An anonymous attacker who knows or obtains a victim family_id (it is non-secret: returned in UserResponse.family_id, embedded in JWT, present in invitation/email links) can POST {"role":"parent","family_id":"<victim>",...} and become a full PARENT inside that family — read/modify the family's budget (bank transactions, receipts), children's data, tasks, points, and approve their own gigs. Complete multi-tenant isolation break.
- Fix: Either delete the /register route entirely (frontend already uses /register-family) or rewrite register_user to ignore client role/family_id: force role from invitation context and require an authenticated PARENT for in-family member creation. Never accept role/family_id from an unauthenticated body. Add a regression test asserting an anonymous POST cannot create a PARENT in an existing family.
- Verify: CONFIRMED. backend/app/api/routes/auth.py:40-53 registers POST /register with no auth dependency (only user_data, request, db). It calls AuthService.register_user (auth_service.py:27-63) which sets role=user_data.role (line 54) and family_id=user_data.family_id (line 55) directly from the client body. Schema UserCreate (schemas/user.py:24-29) exposes role: UserRole = UserRole.CHILD and family_id: UUID as client-controllable. The only guard is auth_service.py:42-47 verifying the family EXISTS — no check that the caller belongs to it or is a parent. Route is registered with prefix /api/auth at main.py:150 and whitelisted public in frontend/src/middleware.ts:76. The backend origin api-gcp-family.agent-ia.mx is publicly reachable via the Cloudflare tunnel (CLAUDE.md). Critically, this raw endpoint has NO legitimate frontend caller: the frontend proxy frontend/src/pages/api/auth/register.ts:53 and parent/members.astro both POST to backend /register-family (the safe self-service path), never the raw /register. test_auth.py:14-25 proves an anonymous POST with arbitrary family_id + role='child' succeeds (201). An attacker can substitute role='parent' and any existing family_id to mint a full-access PARENT in a victim family. Severity critical confirmed — cross-tenant privilege escalation on a live public backend with a fully unused-by-frontend dangerous route.


## [perf]

### [CRITICAL] Month budget dashboard is a deep N+1: ~5 queries per category × N categories per load  — ✅REAL
- Domain: budget · Category: prod-gap · Effort: L
- Evidence: backend/app/api/routes/budget/month.py:121-133 loops `for group in groups: for category in group.categories: summary = await AllocationService.get_category_available_amount(...)`. Each call (backend/app/services/budget/allocation_service.py:253-339) issues: CategoryService.get_by_id (1q) + get_or_create_for_category_month (1+q) + TransactionService.get_category_activity (1q) + 2 rollover SUM queries when rollover_enabled. For a family with 8 groups × 6 categories = 48 categories that is ~240 sequential round-trips for ONE dashboard render, on top of the group fetch and 4 family-wide aggregates at lines 76-90.
- Impact: The main budget dashboard (GET /api/budget/month) — the most-visited budget screen — fans out to hundreds of awaited DB round-trips per load. Latency grows linearly with category count; tail latency on the e2-medium VM will be seconds. This is the single largest perf liability in the budget system.
- Fix: Replace the per-category loop with set-based aggregation: one GROUP BY query for current-month activity per category (SUM(amount) ... WHERE date BETWEEN month AND month_end GROUP BY category_id), one GROUP BY for prior-month rollover budgeted, one for prior activity, and one batch fetch of allocations for the month. Build dicts keyed by category_id, then assemble in Python with zero per-category queries.
- Verify: Confirmed. backend/app/api/routes/budget/month.py:121-135 has nested `for group in groups: for category in group.categories: summary = await AllocationService.get_category_available_amount(db, family_id, category.id, month_date)`. That method (allocation_service.py:252-339) issues: CategoryService.get_by_id (1q at :283), get_or_create_for_category_month (which ITSELF calls CategoryService.get_by_id at :124 plus a SELECT at :134 = 2q), TransactionService.get_category_activity (1q at :352-365), and when rollover_enabled two more SUM queries (:310, :324). That is ~6 sequential round-trips per category, not 5 — finding actually understates it. On top of 4 family-wide aggregates (month.py:76-90) and 1 income query (:118). For 48 categories this is ~290 sequential awaited DB round-trips per single dashboard render. Genuine critical perf hotspot on the primary budget view.

### [CRITICAL] Synchronous OpenAI vision/LLM client used inside async functions with NO timeout — blocks the event loop for up to 600s  — ✅REAL
- Domain: budget · Category: correctness · Effort: 
- Evidence: 6 call sites instantiate the SYNC `OpenAI()` client and call `client.chat.completions.create(...)` directly from `async def` functions with no `timeout=` and no `run_in_executor`/`to_thread`: receipt_scanner_service.py:250-289 (in `async def scan_receipt`), task_proof_validator.py:105-126 (in `async def validate_proof_photo`), frankie_service.py:207-217 and 344-355 (in `async def chat_stream`/`chat`), category_ai_service.py:111-121, calendar_scanner_service.py:93-114, recipe_importer.py:94-105. None pass a timeout, so the OpenAI Python SDK default of 600s applies. Because the call is synchronous, it blocks the entire asyncio event loop (not just one request) until LiteLLM responds. Contrast push_service.py:3 which explicitly notes 'pywebpush is synchronous; wrap each send in asyncio.to_thread' and does so correctly.
- Impact: A slow or hung LiteLLM proxy freezes the whole uvicorn worker for up to 10 minutes per call, stalling ALL concurrent requests (other families' page loads, auth, budget queries). For task_proof_validator this is worst: validate_proof_photo is awaited at task_assignment_service.py:667 while an uncommitted DB transaction is open (commit happens at line 712), so a hung vision call holds a Postgres transaction + connection open for 10 min while freezing the loop. Single biggest availability risk in the codebase.
- Fix: Pass an explicit timeout to every OpenAI client (e.g. `OpenAI(..., timeout=30, max_retries=1)`) AND offload the blocking call with `await asyncio.to_thread(client.chat.completions.create, ...)` (or switch to `AsyncOpenAI` and await it). Mirror the push_service.py to_thread pattern. Move proof validation outside the open DB transaction.
- Verify: VERIFIED across all 6 cited sites. All import the sync client `from openai import OpenAI` (jarvis_service.py:19, task_proof_validator.py:18, calendar_scanner_service.py:19, recipe_importer.py:15, budget/category_ai_service.py:26, budget/receipt_scanner_service.py:33) — none use AsyncOpenAI. Each instantiates `OpenAI(base_url=..., api_key=...)` with NO `timeout=`/`max_retries=` and calls `client.chat.completions.create(...)` synchronously inside an async def: receipt_scanner_service.py:250/275 in `async def scan_receipt` (line 210); task_proof_validator.py:105/114 in `async def validate_proof_photo` (line 77); jarvis_service.py:207/217 in `async def chat_stream` (line 160) and 344/355 in `async def chat` (line 308); category_ai_service.py:111/115 in `async def suggest` (line 66); calendar_scanner_service.py:93/102 in `async def scan_calendar_document` (line 81); recipe_importer.py:94/99 in `async def import_recipe_from_url` (line 72). grep confirms zero `to_thread`/`run_in_executor`/AsyncOpenAI in these files; the httpx timeouts present (task_proof_validator.py:70, recipe_importer.py:83) are for separate image/URL fetches, not the LLM call. No global OpenAI timeout config exists in app/ or config.py, so the SDK 600s default applies. The sync call blocks the whole asyncio event loop. Contrast push_service.py:3 documents 'pywebpush is synchronous; wrap each send in asyncio.to_thread' and does so at line 106. Note: 'frankie_service.py' was renamed to jarvis_service.py (rename in progress) but the cited line numbers and issue are identical.

### [HIGH] No rate limiting on any endpoint — auth (login, register, check-methods, forgot-password) open to brute force and enumeration  — ✅REAL
- Domain: auth · Category: security · Effort: M
- Evidence: Grep across backend/app and requirements.txt for slowapi/Limiter/rate_limit/RateLimiter returns nothing — there is no limiter middleware or dependency anywhere. backend/app/api/routes/auth.py exposes /login (line 155), /check-methods (178, an explicit account-existence oracle returning has_password/has_google), /forgot-password (308), /register (43), all unauthenticated. bcrypt verify is the only cost per login attempt.
- Impact: Credential stuffing / password brute force against /login, and account enumeration via /check-methods (directly reveals whether an email is registered and which auth methods it has) at unbounded request rate. - Fix: Add a Redis-backed sliding-window limiter (e.g. slowapi or a small custom dependency keyed on IP + email) on /login, /register(+family), /check-methods, /forgot-password, /reset-password. Add progressive backoff / temporary lockout after N failed logins per account.
- Verify: CONFIRMED. grep across backend/app and requirements.txt for slowapi/limiter/rate.?limit/ratelimit returns no limiter library or middleware — the only hits are an unrelated push-notification 'Rate limit' comment (notification_service.py:48) and 'rate limits' in a receipt-scanner error comment. main.py adds only CORSMiddleware and SessionMiddleware (lines 124,133) — no rate-limit middleware. Unauthenticated auth endpoints confirmed: /login (auth.py:155), /register (43), /forgot-password (308), /check-methods (178). bcrypt verify (security.py:17-19) is the only per-attempt cost on login. /check-methods (auth.py:178-204) is an explicit account-existence + auth-method oracle: returns has_password/has_google for known emails and {false,false} for unknown, distinguishing registered emails. /forgot-password (313-321) correctly returns a constant response to avoid enumeration, but is still unthrottled. With no limiter, online password brute-force, account enumeration via check-methods, and email-bombing via forgot-password are all unmitigated. ### [HIGH] Double-award / double-trust-streak race on gig approval — check-then-write with no row lock or idempotency key  — ✅REAL
- Domain: gig · Category: correctness · Effort: M
- Evidence: GigClaimService.approve reads status then awards in a separate commit with no FOR UPDATE: backend/app/services/gig_claim_service.py:312-338 (`if claim.status != GigClaimStatus.COMPLETED: raise ...` then `claimer.points += points; claimer.gig_trust_streak += 1; ... await db.commit()`).
