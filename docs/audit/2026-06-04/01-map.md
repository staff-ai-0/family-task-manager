# Codebase Map (Phase 1 — 8 cluster explorers)

## AUTH/CORE

The AUTH/CORE cluster handles user authentication, family management, premium subscriptions (PayPal), user invitations, email verification, password resets, Google OAuth, web push notifications, and in-app notifications. It uses JWT tokens for auth, bcrypt for passwords, family-based access control (all users belong to exactly one family), and integrates PayPal v2 Billing Subscriptions API. Notifications support both in-app feed and web-push delivery to parent devices. Email is sent via SMTP (Google Workspace) or Resend API with bilingual templates. The cluster is mature with comprehensive authorization checks but has some debt around missing cascade deletes and incomplete secrets validation.

**Debt smells:** No rate limiting on auth endpoints (register, login, check-methods, forgot-password) — vulnerable to brute force and enumeration at scale. Consider redis-based sliding window limiter.; Password reset and email verification tokens stored in DB with simple UUID expiry — no rotation or one-time-use enforcement after consumption (token.consumed_at set but not checked on re-use).; OAuth account linking doesn't support multi-account linking or unlinking UI — if user has both password + OAuth, confusion on which method to use.; PayPal webhook signature verification depends on PAYPAL_WEBHOOK_ID env var; if missing, endpoint raises 401 for all webhooks rather than failing gracefully or warning at startup.; Family deletion cascades via SQLAlchemy but some related records (task_templates, task_assignments, budgets) have complex cascade rules — untested large-scale cascade scenarios.; Email failures (SMTP/Resend) are swallowed with logging; no alerting on persistent email delivery failures (e.g. invalid SMTP creds).; User.gig_trust_streak field auto-approval logic not fully audited — if parent rejects on edge case, could cause state divergence.; Invitation acceptance doesn't validate that only PARENT role can accept (currently allows all roles, then gating check happens post-creation).; Premium feature gates (family_member quota) checked before InvitationService.send but not in a transaction — race condition if two rapid requests slip through.; No secrets rotation policy for JWT SECRET_KEY, PAYPAL_CLIENT_SECRET, SMTP_PASSWORD — long-lived secrets in .env or Vault.; CORS configured with allow_origins from settings, but no validation that PUBLIC_URL/BASE_URL are in the allow_list — email links could redirect to attacker domain.; No audit logging of sensitive operations (user deletion, role changes, invitation acceptance) — compliance and forensics gaps.; User.email_verified field not explicitly checked on login (deactivation, account disabled, or email not verified doesn't block login).; Session middleware secret_key uses same SECRET_KEY as JWT — cryptographic key reuse antipattern.; OAuth email auto-verification trusts Google's email_verified field without re-validating ownership — medium risk if Google account hijacked later.; Invitation email includes raw invitation code in URL; code format (24-char url-safe) suitable for timing-attack resistance but not cryptographically analyzed.; No CSRF protection on state-changing endpoints (POST, PUT, DELETE) beyond CORS — SOP-based only.

The system manages a complex lifecycle: templates→weekly shuffle→assignment→completion/claiming→mandatory vs. bonus gating→approval→points award, plus auto-penalty consequences on overdue tasks.

### Shuffle Algorithm (Weekly Generation)
- Purpose: Deterministic mapping from templates→assignments. Seeds RNG by family_id:week_monday for idempotency. Handles AUTO (load-balance), FIXED (same user), ROTATE (cycle users), bonus modes (claim/rotation/competition/collaboration).
- Files: /Volumes/shared/AgentIA/family-task-manager/backend/app/services/task_assignment_service.py (lines 54-406)
- Endpoints: 2 (e.g. POST /api/task-assignments/shuffle; GET /api/task-assignments/shuffle/preview)
- Notes: Pure builder: _compute_assignments() generates list of TaskAssignment ORM objects without DB writes. Caller (shuffle_tasks) does add/commit. Input: regular_templates (is_bonus=False), bonus_templates (is_bonus=True), active members. Cross-week carry: sum points from prior 2 weeks, members with high carry are picked last for fairness. Day-load cache: for each (member, date) pair, track accumulated points. FIXED: always assign to first eligible user. ROTATE: cycle through eligible list, incrementing per assignment. AUTO interval_days=7: pick (member, day) with min(day_load + cross_week_bias). AUTO interval_days=1: pick member with min load for EACH day. AUTO else: pick member with min cumulative load across all dates + max daily impact. Bonus expansion: interval_days maps to date list. CLAIM/COLLABORATION: all eligible members get row per date. ROTATION: cycle members by week index. Returns (assignments, totals_per_member) where totals exclude carry.
