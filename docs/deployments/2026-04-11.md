## Known gap flagged but deliberately NOT touched

The active `docker-compose.yml` runs backend with `uvicorn --reload` and `DEBUG=${DEBUG:-true}`, and frontend with `npm run dev --host` (Astro dev server with HMR). Both services bind-mount `./backend/app` and `./frontend` from the host into the containers, meaning a `git pull` effectively hot-swaps live code without a container restart. This is functional but is a **development configuration running in production**.

A proper hardening pass would need:

- A multi-stage frontend `Dockerfile` that runs `npm run build` at image build time and serves the built SSR entry (e.g. `node dist/server/entry.mjs`) at runtime.
- A `docker-compose.prod.yml` override flipping the backend command to `uvicorn --host 0.0.0.0 --port 8000` (no `--reload`), setting `DEBUG=false`, and dropping the host bind mounts.
- End-to-end smoke tests of the built SSR entry against auth middleware, i18n routing, env var propagation, and the budget pages.

This is a meaningful change with its own test risk, out of scope for today's deploy. Deferred as a separate follow-up.

## Google OAuth login status

Investigation triggered by a user report of "Google login not working".
Phase 1 gathered the following evidence before any fix was attempted:

- Backend route `/api/oauth/google`: reachable, validator runs correctly (returns `401 "Wrong number of segments in token"` on bad tokens).
- Astro proxy `/api/oauth/google`: forwards to backend via `http://backend:8000` (docker compose service DNS).
- Env: `GOOGLE_CLIENT_ID` matches between backend and the hardcoded value in `frontend/src/pages/login.astro` (`436801934159-r2klrd5n7b4da2q806c58utd24h9f71v.apps.googleusercontent.com`), `GOOGLE_CLIENT_SECRET` set.
- Public URL is fronted by Cloudflare, no CSP/COOP headers blocking the GIS popup.
- Logs showed zero `/api/oauth/google` hits from the user's browser — only from CLI test curls — meaning the failure was 100% client-side (the click wasn't producing a request that reached our server).

No speculative fix was made against this code path, because the root cause wasn't isolated without the browser console output. Subsequently the user reported back that their account `juan.mtz79@gmail.com` was already successfully created via Google OAuth (verified via DB: `oauth_provider=google` with populated `oauth_id`). So the flow resolved itself between the initial report and the follow-up. Possible causes: origin config propagated in GCP, FedCM consent state cleared, or the initial failure was for a reason we never captured. Not chased further.

The new `check-methods` endpoint + login-form redirect means that even if Google login were broken again in the future, typing `juan.mtz79@gmail.com` into the email field would trigger the GIS sign-in flow with `login_hint` pre-populated, giving users a reasonable failure mode.
